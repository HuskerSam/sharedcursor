The download here (with 2 different custom avatars loaded) is ~70mb. The first 6 large items are the avatars which also contain bone animation sequences.

Starting from item 7 on it's textures and static assets for the environment. Containing the size is important for download performance and containing the polygon count is important for the GPU performance (Quest 2 supports ~2 million). Using the debugging tools in the webKit browsers makes it easy to identify the largest assets to refactor. Meshes can be decimated to reduce size and polygon count, but most of the size reductions for 3rd party assets come from converting internal images from png to jpg.

Saving about 25% of GPU performance is required if you plan to stream video to external devices (common user activity).

The Apollo Spacesuit and LK Lander are licensed only for this site, but feel free to harvest the other asset files as they're derived from publicly available sources. If in doubt find the public source then before using it.

UV Maps for asteroid meshes


Spherical projection UV maps are generated in Blender and saved out in a new .obj file (best for size) for asteroids. This allows a common texture to be used for all minor asteroids.

Query string asteroidcount can be used to adjust the number of asteroids in the environment, 15 is the default, currently there are 234 small asteroids available. Near Earth objects and asteroids with symbols assigned are treated like a moon asset.

Crowd Navigation ▼

BabylonJS supports Recast crowd navigation. Assets can supply a navRadius to be included as an obstable that avatars must navigate around when moving.

The floor and obstacles are shown here as hot pink - note only a disc is used to block the area that avatars (agents) navigate through.
When avatars, probes and rockets move around agents are used that adjust pathing on the fly to adjust to other moving obstacles.

Event Playback ▼

Goals Multiuser game document data - live, replayable and user action driven.
Clear NPC and player narration and interaction in a shared envinronment.
Notes Users get realtime updates, but the timing of the current user environment may not be live/synced - the channels merge the difference between action and speaking queues and user actions in realtime. If both users are viewing the current round and both environments have played all actions then new actions will occur in realtime, otherwise they are queued to happen when possible (turn based game with smooth playback).

Action and Speech Channels▼

The Action Channel is the primary animation queue
The Speech Channel is the narration queue with additive animations (~emotes)
Stored round data contains events for both channels
Can be replayed at different times on users devices
Events can optionally block until the next event


GPU mesh morping to support bone animated avatars is our main load on the GPU currently.
Limiting the number of animated avatars is critical to save GPU compute for other tasks.
So there is an action channel; this is the main animation channel, and a speech channel for users to interact. In practice for a turn based game, the active player controls the active channel, and the speech channel is open to all players. The speech channel blends in animations per event - that can either stand alone or combine with a walking animation. For example in the speech channel an add animation can be specific as throw up arms - this will combine with walking animation if the avatar is moving, or will run on it's own if the avatar is standing still. Chat is broken down into fragments - sentances as best as possible and displayed on both cartoon style popups and google text to speech is used to create audio.
In addtion to speech and action channel events a round contains and starting asset positions if different from default

The default round is the intial setup - and each round is a delta against it, for example for round 5 you need the record for round 5 and the default round, but none of the other rounds to determine current state. This provides for a large number of rounds (and data) but keeps the initial load data size small.
When a new round is created assets not in the default position are specified with state change items

prequel rounds (vs game rounds created by users) can contain premade story events and/or tutorials on game play
